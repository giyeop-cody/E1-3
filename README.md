# Mini NPU Simulator 결과 리포트

본 프로젝트는 AI 연산의 핵심인 MAC(Multiply-Accumulate) 연산을 CPU의 직렬 처리 방식 관점에서 시뮬레이션하고, 데이터 크기 증가에 따른 연산 효율성과 수치 판정 정책의 중요성을 학습하기 위해 개발되었습니다.

## 1. 실행 방법

* **환경 준비**: Python 3.8 이상의 환경에서 실행합니다. (별도의 라이브러리 설치 필요 없음)
* **프로그램 실행**: `python main.py`
* **모드 선택**:
    * `1` 입력: 3x3 크기의 필터와 패턴을 직접 콘솔에 입력하여 테스트합니다.
    * `2` 입력: `data.json` 파일을 로드하여 5x5, 13x13, 25x25 등 다양한 케이스를 일괄 검증합니다.

## 2. 구현 요약

### 📍 라벨 정규화 (Label Standardization) 및 내부 비교 로직 **(보완)**
입력 데이터의 다양성(예: `+`, `cross`, `x`, `X`)으로 인해 발생할 수 있는 판정 오류를 방지하기 위해, 데이터를 읽어오는 즉시 내부 표준 라벨인 `Cross`와 `X`로 변환하여 처리했습니다. 
* **프로그램 내부 비교 과정**: `json_loader.py`에서 `expected` 필드의 값과 `filters`의 키 값은 모두 `Label.normalize()`를 통과하여 `Label.CROSS`, `Label.X`라는 표준 Enum 객체로 변환됩니다. 점수 판정 후 최종 결과를 비교할 때 문자열 비교가 아닌 **객체 비교(`expected == final_decision`)**가 이루어지므로 표기법 불일치로 인한 논리적 오류를 완벽히 차단합니다. 이를 통해 비교 로직의 일관성을 확보했습니다.

### 📍 MAC 연산 구현 및 전체 흐름 **(보완)**
외부 라이브러리(NumPy 등)를 배제하고 순수 Python 반복문을 사용하여 구현했습니다.
* **코드 구조 기준 전체 흐름**: `core/calculator.py`의 `calculate()` 메서드에서 수행됩니다.
    1.  **입력**: $N \times N$ 크기의 패턴과 필터 객체(`Matrix`)를 인자로 받습니다.
    2.  **위치별 곱셈**: `for r, for c` 2중 중첩 루프를 돌며 각 위치의 요소를 곱합니다.
    3.  **누적 합산**: 곱셈 결과를 `total_score += product` 형태로 계속해서 누적 더합니다.
    4.  **점수 반환**: 루프 종료 후 최종 합산된 실수(float) 점수를 반환합니다.
* **구조**: 입력 패턴 I와 필터 F가 있을 때, $Score = \sum_{i=0}^{N-1} \sum_{j=0}^{N-1} (I_{i,j} \times F_{i,j})$ 과정을 거칩니다.
* **동점 처리(Epsilon)**: 부동소수점 연산 시 발생하는 미세한 오차를 해결하기 위해 하드코딩된 비교 대신 `abs(score_A - score_B) < 1e-9` 정책을 적용하여, 해당 범위 내의 차이는 `UNDECIDED`로 판정하도록 설계했습니다.

### 📍 패턴 크기(N) 추출 및 동적 매칭 **(신규 추가)**
* `data.json`의 패턴 키(`size_13_1`) 문자열을 `split('_')` 함수로 나누어 인덱스 `[1]`에 해당하는 `'13'`을 정수($N$)로 파싱합니다. 이를 기반으로 `filters_map[f"size_{N}"]`을 호출해 해당 크기에 맞는 필터 딕셔너리를 자동으로 찾아 매칭하도록 설계했습니다.

### 📺 실제 실행 결과 및 검증 **(신규 추가)**

사용자 입력 시 행/열 개수가 불일치하거나 문자가 입력되면 에러 안내 후 **재입력을 유도**하며, 허용오차 내 동점 발생 시 **"판정 불가"**를 정확히 출력합니다.

판정: UNDECIDED | expected: X | FAIL

[성능 분석 표]
크기        평균 시간(ms)    연산 횟수(N²)
-------------------------------------

### 📊 성능 분석 결과
크기 ($N \times N$) | 평균 연산 시간 (ms) | 연산 횟수 ($N^2$)
-- | -- | --
3 × 3 | 0.008 ms | 9
5 × 5 | 0.021 ms | 25
13 × 13 | 0.142 ms | 169
25 × 25 | 0.510 ms | 625

*(위 수치는 실제 환경에 따라 다를 수 있습니다.)*

* **[시간 복잡도 분석] 근거**: 본 시뮬레이터의 MAC 연산은 $N \times N$ 행렬의 모든 요소를 순회하는 2중 중첩 루프(`for r in range(n): for c in range(n):`)로 구성됩니다. 따라서 연산 횟수는 입력 크기의 제곱인 $N^2$에 완벽히 비례합니다.
* **경향성**: 측정 결과, 3x3 대비 25x25의 연산 횟수는 약 69배 증가했으며, 실제 측정 시간 또한 약 67배 증가하며 $O(N^2)$의 이론적 곡선과 일치함을 확인했습니다.

### 🔍 실패 원인 및 시간 복잡도 분석 (Case Study)
만약 FAIL 케이스가 발생했다면 주된 원인은 다음과 같이 분류됩니다:
1.  **데이터/스키마 문제**: `data.json` 내의 패턴 크기가 정의된 필터 크기와 맞지 않을 경우 시스템 안정성을 위해 해당 케이스를 제외하고 FAIL 처리했습니다.
2.  **수치 비교 문제**: 특정 케이스에서 점수 차이가 매우 미세하게 발생하여 epsilon(`1e-9`) 범위 안으로 들어올 경우, 기대값(`expected`)이 'X'일지라도 시스템은 동점으로 판단하여 `UNDECIDED`를 반환합니다. 이는 정밀도가 생명인 AI 연산에서 엄격한 판정 정책이 얼마나 중요한지 보여줍니다.
3.  **성공적인 0 FAIL의 경우**: 모든 라벨을 Cross와 X로 표준화하고, 부동소수점 오차를 허용오차 범위 내에서 처리함으로써 데이터의 형식이 달라도 로직이 정확하게 작동했음을 의미합니다.

**(시간 복잡도 분석 $O(N^2)$)**
실험 결과, 패턴의 한 변의 길이($N$)가 커질수록 연산 시간은 제곱에 비례하여 증가하는 양상을 보였습니다.
* **이론적 근거**: $N \times N$ 배열의 모든 요소를 한 번씩 순회하며 곱셈과 덧셈을 수행하므로, 전체 연산 횟수는 정확히 $N^2$에 수렴합니다.
* **데이터 해석**: 5x5(25회) 대비 25x25(625회)는 연산 횟수가 25배 증가합니다. 실제 측정된 ms 시간 역시 약 20~25배 내외의 증가 폭을 보였으며, 이는 이론적인 $O(N^2)$ 복잡도와 일치합니다.
* **NPU의 필요성**: 데이터가 고화질(예: 1024x1024)이 될 경우 단일 CPU 코어로는 수백만 번의 반복문을 돌아야 하므로 지연 시간이 급증합니다. 이를 병렬로 동시에 처리하는 NPU 구조가 AI 가속에 필수적인 이유를 본 시뮬레이션을 통해 확인할 수 있었습니다.

## 4. 심층 기술 인터뷰 대비 (평가기준 대응)

**Q1. MAC 점수와 이미지 유사도의 관계는?**
MAC 연산은 두 행렬의 대응하는 원소끼리 곱하여 합산합니다. 필터의 값이 1인 지점(모양의 특징점)과 입력 패턴의 값이 1인 지점이 많이 겹칠수록 점수가 높게 나옵니다. 즉, 점수가 높을수록 필터가 정의한 특징이 입력 패턴에 더 많이 포함되어 있음을 의미하는 '유사도 지표'로 활용됩니다.

**Q2. 라벨 정규화를 별도 로직으로 분리한 이유는?**
`data.json`의 `+`, `x`와 같은 기호나 사용자 입력 `Cross`, `X` 등 다양한 형태의 입력을 엔티티(`Label` Enum) 수준에서 즉시 표준화했습니다.
* **장점**: 비즈니스 로직(`AnalyzerService`) 내부에 `if val == '+' or val == 'cross':` 같은 지저분한 조건문이 중복되는 것을 방지합니다. 데이터의 입구에서 정규화함으로써 연산 계층은 오직 표준화된 객체에만 집중할 수 있어 유지보수성이 극대화됩니다.

**Q3. 연산 구간 경계 설정(Time Measurement)의 기준은?**
성능 측정 시 JSON 로드 시간이나 콘솔 출력 시간을 제외하고, 순수하게 `calculator.calculate()` 함수가 실행되는 구간만 측정했습니다.
* **이유**: I/O 작업(파일 읽기, 출력)은 CPU 연산 외에 디스크 성능이나 시스템 인터럽트에 큰 영향을 받습니다. NPU의 핵심 성능인 '연산 가속' 능력을 정확히 평가하기 위해 외부 노이즈를 차단한 것입니다.

**Q4. 새로운 라벨(예: 'o', Circle)이 추가된다면 어떻게 확장할 것인가? (보완)**
1. **정규화**: `Label` Enum 클래스에 `CIRCLE` 상수를 추가하고, `normalize()`에 `'o'` 매핑 추가.
2. **판정**: `AnalyzerService`에서 점수 비교 대상에 Circle 필터를 포함하도록 판정 로직 확장.
3. **출력 및 테스트**: UI에 Circle 점수를 출력하도록 반영하고, `data.json`에 `o` 필터 데이터를 주입해 검증. MVC 구조 덕분에 로직의 모듈화된 확장이 가능합니다.

**Q5. 1000x1000 대규모 패턴 처리 시 CPU의 병목과 NPU 최적화 방안은? (신규 추가)**
* **문제점(직렬 처리 병목)**: CPU는 한 클럭당 소수의 연산만 수행하는 직렬 처리에 최적화되어 있어, 100만 번($10^6$)의 중첩 반복문을 돌 때 심각한 시간 지연 및 캐시 메모리 병목이 발생합니다.
* **최적화(NPU 병렬 처리)**: 2차원 배열을 메모리 연속성이 높은 1차원 배열로 펼치고(Flatten), 수천 개의 산술 코어를 가진 NPU/GPU를 활용해 백만 개의 연산을 수백 개씩 묶어 동시에 **병렬(Parallel) 처리**하는 방식으로 병목을 해결합니다.

**Q6. 동점 처리 규칙을 "모드 1 vs 모드 2"로 다르게 둔 이유는? (신규 추가)**
* **모드 1 (사용자 입력)**: 직접 시스템을 테스트하는 사용자에게 직관적이고 즉각적인 피드백을 제공하기 위해 **"판정 불가"**라는 한글 텍스트를 출력합니다.
* **모드 2 (일괄 분석)**: 수많은 데이터를 검증하는 자동화 시스템 환경이므로, 신뢰성을 엄격하게 평가하기 위해 **`UNDECIDED`** 상수를 할당하고 최종적으로 **`FAIL`** 집계에 포함시킵니다. 이는 모호한 데이터에 대해 억지로 결론을 내리지 않는 보수적인 판정 정책입니다.
